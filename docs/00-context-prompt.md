This document is the single source of truth for the BioVibe codebase. It is referenced by all implementation prompts via `@docs/00-context-prompt.md` so the Cursor Agent has full context before starting any task.

---

## What is BioVibe

BioVibe is a WhatsApp conversational agent that acts as a personal biohacking assistant. Users send voice notes or text messages about meals, symptoms, sleep, mood, and activity. The AI logs everything in a structured JSON file (one per user), identifies patterns, and replies with a proactive health insight after every message.

Full product specification: `docs/PRD-BioVibe.md`

## How Messages Arrive

Two message types come from the Kapso webhook. `inbound_text(msg)` in `bot.py` already handles both and returns a plain string:

| Type | Source field | Example log |
|---|---|---|
| `text` | `msg.text.body` | `INBOUND \| from=5521967010022 type=text message='Oi tudo bem?'` |
| `audio` | `msg.kapso.content` (Kapso transcribes automatically) | `INBOUND \| from=5521967010022 type=audio message='Audio attached... Transcript: Oi, tudo bem?'` |

`msg.type` (`"text"` or `"audio"`) must be preserved as `media_type` in any log entry saved to the Brain.

---

## Intent Model

Every message is classified by Gemini into one of four intents:

| Intent | Meaning | Brain write? | Implemented in |
|---|---|---|---|
| `log` | User is recording health data | Yes — append to `log_history` | prompt 02 |
| `query` | Health-related question or summary request | No | prompt 02 |
| `unrecognized` | Unrelated to health tracking | No — reply with friendly onboarding message | prompt 02 |
| `profile_update` | User shares persistent personal info (name, allergies, traits) | Yes — update `profile` only | prompt 05 |

## User Brain File Schema

File path: `data/{user_id}.json` where `user_id = msg.phone_number` (e.g. `"5521967010022"`).

```json
{
  "user_id": "5521967010022",
  "profile": {
    "name": null,
    "traits": []
  },
  "health_summary": "",
  "log_history": [
    {
      "id": "<uuid4>",
      "timestamp": "<ISO8601 UTC>",
      "category": "Nutrition | Symptom | Activity | Sleep | Mood",
      "raw_input": "<original message text>",
      "media_type": "text | audio",
      "structured": {}
    }
  ]
}
```

`health_summary` is regenerated by Gemini every 5 new log entries.

## Message Flow

```
inbound_text(msg) → plain text (same for text and audio)
        │
        ▼
Gemini (text + brain context)
        │
        ▼
┌────────── Intent ──────────┐
│        │        │          │
LOG    QUERY  UNRECOGNIZED  PROFILE_UPDATE
│        │        │          │  (prompt 05)
▼        ▼        ▼          ▼
Save   Read   Friendly   Update profile
Brain  Brain  reply only  (name + traits)
│        │                   │
▼        │                   │
Reply ◄──┴───────────────────┘
│
▼
client.send_whatsapp_message()
```

## Environment

- Ngrok tunnel: `ngrok http 8000`
- Webhook URL in Kapso: `https://<ngrok-host>/webhooks/whatsapp`
- Verify token: `biovibe-verify-2026`

## Coding Conventions

- `gemini_client.py` and `kapso_client.py` are `async` — follow the pattern in `kapso_client.py`
- `brain.py` is synchronous — file I/O only, no network calls
- `bot.py` is the orchestrator — business logic belongs in `services/`
- Use `logging.getLogger(__name__)` in every module
- Plain text only in WhatsApp replies — no markdown
- Use `pathlib.Path` for all file paths — no `os.path`
- No comments that just describe what the code does — only explain non-obvious intent
